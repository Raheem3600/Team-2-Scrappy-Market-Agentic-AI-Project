# Scrappy Market Agentic AI – System Architecture

## Overview

The Scrappy Market Agentic AI platform enables business users to investigate retail performance using natural language questions. The system uses a multi-agent architecture powered by LangGraph, a FastAPI backend that provides controlled access to the data layer, and a SQL Server database containing synthetic retail datasets and analytical views.

This architecture is designed to support explainable AI investigations while ensuring safe, structured interaction with the database.

---

# High-Level System Flow

User → Streamlit UI → LangGraph Agents → FastAPI API Layer → SQL Server → Response Agent → Streamlit UI

```mermaid
## Streamlit UI Layer

Streamlit acts as the user interface layer of the system. It allows business users to interact with the system using natural language.

Responsibilities

Streamlit is responsible for:

• Capturing the user’s investigation question
• Displaying generated SQL queries
• Showing query results in tables or visualizations
• Displaying reasoning explanations generated by agents
• Showing investigation confidence scores

Example UI Interaction

User question:

Why did sales drop in the Mountain region last quarter?

Streamlit may display:

Generated SQL

SELECT Region,
       SUM(SalesAmount) AS total_sales
FROM vw_sales_enriched
WHERE Region = 'Mountain'
AND Quarter = 4
GROUP BY Region

Results

Region	Sales
Mountain	$1.2M

Explanation

Sales declined due to reduced promotional activity and lower product demand in the Mountain region during Q4.

Confidence Score

0.82

Displaying the SQL query helps provide transparency into how the AI system derived its answer.

## LangGraph Orchestration Layer

---

LangGraph coordinates the multi-agent investigation workflow and maintains the Investigation State.

The investigation state may include:

• identified intent
• generated hypotheses
• supporting evidence
• confidence score

This orchestration allows agents to reason step-by-step rather than jumping directly to a database query.

---

## Agent Workflow

---
The investigation pipeline consists of several specialized agents.

Intent Agent

The Intent Agent analyzes the user’s question and extracts key investigation elements.

Example extraction:

Metric: SalesAmount
Dimension: Region
Filter: Mountain
Time Range: Last Quarter

Investigation Planner Agent

The Planner Agent determines how to investigate the question by generating a step-by-step investigation plan.

Example plan:

Compare current quarter sales to previous quarter

Analyze promotion activity

Identify product category performance

Lineage / Context Agent

The Lineage Agent discovers database structure dynamically using metadata APIs.

Example API calls:

GET /meta/views
GET /meta/columns

This allows agents to understand:

• available analytical views
• fields available in each view
• relationships between datasets

Query Builder Agent

The Query Builder Agent generates SQL queries based on the investigation plan and schema metadata.

Example generated SQL:

SELECT Region,
       SUM(SalesAmount) AS total_sales
FROM vw_sales_enriched
WHERE Region = 'Mountain'
AND Quarter = 4
GROUP BY Region
Data Validation Agent

Before execution, the query is validated to ensure:

• it is syntactically correct
• it matches the investigation plan
• it does not contain unsafe operations

Response Agent

After query results are returned, the Response Agent generates a natural-language explanation for the user.

Example:

Sales declined by 18% in the Mountain region compared to the previous quarter due to reduced promotional activity and lower dairy product sales.

The response also includes a confidence score indicating how reliable the explanation is.

## FastAPI Backend API Layer

---

The FastAPI backend acts as the controlled gateway between agents and the database.

Agents do not connect directly to SQL Server. Instead, they interact with the database through API endpoints.

Metadata Endpoints

These endpoints allow agents to discover database structure dynamically.

GET /meta/views
GET /meta/columns
Query Execution Endpoint
POST /query/execute

The API is responsible for:

• exposing schema metadata to agents
• validating incoming SQL queries
• executing queries against the database
• returning structured JSON responses

## SQL Server Data Layer
```
---
The system uses a SQL Server database containing the ScrappyMarket schema.

Core Tables

• Sales
• Products
• Stores
• Dates
• Promotions
• PromotionProducts
• Inventory
• Users

These tables store synthetic retail data used for investigation scenarios.

Semantic Views

Agents interact primarily with semantic analytical views instead of raw tables.

Examples:

• vw_sales_enriched
• vw_sales_daily_store
• vw_sales_daily_product
• vw_promotions_enriched
• vw_low_stock

Using views simplifies queries and reduces complexity for agents.

## Deployment Architecture

---
The system is designed to be deployed using Docker containers.

Container 1 – Application Layer

Contains:

• Streamlit UI
• LangGraph Agents
• FastAPI backend service

Container 2 – Data Layer

Contains:

• SQL Server
• ScrappyMarket database schema
• synthetic retail datasets

The containers communicate through internal networking.
